01 — Introducción

¿Por qué hablar de seguridad hoy?

¿Alguna vez escuchaste que es mejor prevenir que curar?
En el desarrollo de software, esa frase cobra un significado muy especial, y es exactamente de lo que vamos a hablar hoy.

Somos Paulina Chica y Antonio Mendieta, estudiantes de Ingeniería de Software e integrantes del Grupo #8. Como futuros ingenieros, sabemos que escribir código es solo una parte del trabajo. La otra parte, que muchas veces se ignora hasta que ya es tarde, es asegurarnos de que ese código sea seguro desde el primer día.

Imagina esto: un equipo trabaja durante meses en una aplicación, la lanza al mercado, y una semana después descubren que cualquier persona puede acceder a la información privada de los usuarios con un simple truco en la URL. ¿El resultado? Pérdida de datos, usuarios afectados, reputación dañada y un costo enorme para arreglarlo.Todo eso pudo haberse evitado si la seguridad se hubiera pensado desde el inicio.

Eso es exactamente lo que propone Shift Left Security: mover las pruebas y controles de seguridad hacia las etapas más tempranas del desarrollo. Mientras más a la izquierda del ciclo de vida se detecta un problema, más fácil y barato es corregirlo.

A continuación te explicamos qué es Shift Left Security, por qué importa y cómo cambia la forma en que se construye software seguro.

02 — Definición

¿Qué es Shift Left Security?

(OWASP, 2025)

Shift Left Security es la práctica de integrar la seguridad en las primeras etapas del ciclo de vida del desarrollo de software (SDLC), en lugar de tratarla como una fase separada al final del proceso.

(NIST, 2024)

Shifting left significa mover la seguridad más temprano en el pipeline de desarrollo, integrándola en las fases de diseño, codificación y pruebas, para que las vulnerabilidades se detecten antes de llegar a producción.

Básicamente, en el desarrollo tradicional la seguridad siempre se dejaba para el final, lo que terminaba siendo un dolor de cabeza porque arreglar los fallos a última hora retrasaba todo el despliegue. Al aplicar Shift Left, lo que hacemos es cambiar esa mentalidad: metemos la seguridad desde el día uno en el código y en cada Sprint, haciendo pruebas continuas para atrapar los errores rápido, ahorrar recursos y lanzar software que ya nace seguro.

Los 5 Principios

1
Seguridad desde el diseño

Antes de escribir una sola línea de código, el equipo debe identificar amenazas y definir cómo mitigarlas. La seguridad es parte de la arquitectura, no un parche posterior.

2
Integración continua de seguridad

Las pruebas de seguridad se automatizan e integran en el pipeline CI/CD. Cada vez que se sube código, se analiza automáticamente en busca de vulnerabilidades.

3
Responsabilidad compartida

La seguridad no es solo tarea del equipo de ciberseguridad — es responsabilidad de todos: desarrolladores, testers, diseñadores y líderes de proyecto.

4
Detección temprana de vulnerabilidades

Corregir un fallo en producción cuesta hasta 100 veces más que detectarlo a tiempo en la fase de desarrollo. El uso de herramientas automatizadas como SAST y DAST durante el sprint evita brechas críticas y optimiza los costos de infraestructura.

5
Cultura y formación continua

El equipo se capacita continuamente en buenas prácticas. Un desarrollador que conoce el OWASP Top 10 escribe código más seguro desde el inicio.

El pipeline del SDLC: tradicional vs Shift Left

En el enfoque tradicional la seguridad llega al final, cuando corregir un fallo ya cuesta hasta 100 veces más. Shift Left la integra desde los requisitos, distribuyendo el esfuerzo a lo largo de todo el ciclo de vida.

03 — Problema que soluciona

El costo de ignorar la seguridad

Costos altísimos de corrección

Arreglar una vulnerabilidad en producción puede costar hasta 100 veces más que detectarla durante el desarrollo.

Retrasos en los lanzamientos

Cuando se descubren fallos al final, todo el proyecto se detiene. Los sprints se alargan y los plazos se incumplen.

Datos de usuarios en riesgo

Las brechas exponen información sensible, generando pérdidas de confianza, sanciones legales y daño reputacional.

Deuda técnica acumulada

El código inseguro se acumula sprint tras sprint, volviéndose cada vez más difícil y costoso de corregir.

Equipos desconectados

Los desarrolladores escriben código y el equipo de seguridad lo revisa semanas después, sin cultura compartida.

La solución: Shift Left

Integrar seguridad desde el inicio reduce todos estos problemas, construyendo software más robusto y equipos más conscientes.

04 — Metodología Ágil

Relación con Sprints Ágiles

En metodologías ágiles, Shift Left Security propone que cada Sprint tenga sus propias actividades de seguridad integradas. Un Sprint no está completo si el código no pasó las pruebas de seguridad correspondientes.

Sprint Actividad Seguridad aplicada
Sprint 1 Login Validación de entrada, protección contra SQL Injection
Sprint 2 Registro de usuarios Hash de contraseñas, validación de formularios
Sprint 3 Gestión de sesiones Tokens seguros, expiración de sesiones
Sprint 4 Subida de archivos Validación de tipo/tamaño, sanitización de rutas
Sprint 5 Panel de administración Control de acceso por roles (RBAC), 2FA

Código vulnerable y su corrección

El siguiente fragmento de código corresponde al grupo #8. A simple vista parece funcional, pero contiene vulnerabilidades reales que en producción podrían comprometer la estabilidad del sistema.

VulnerableServlet.java ⚠ VULNERABLE 
// Código del Grupo #8 — SIN aplicar Shift Left Security

String edadStr = request.getParameter("edad");
int edad = Integer.parseInt(edadStr);     // ⚠ Sin validar null ni formato

if (edad > 18) {
    registrarUsuario();
}

Código mejorado — Shift Left aplicado

Cada bloque del código corrige directamente una de las vulnerabilidades identificadas en la tabla anterior.

SecureServlet.java ✓ SEGURO 
// Código del Grupo #8 — CON Shift Left Security aplicado

String edadStr = request.getParameter("edad");

// 1. Validar que el parámetro no sea null o vacío
if (edadStr == null || edadStr.trim().isEmpty()) {
    response.sendError(HttpServletResponse.SC_BAD_REQUEST, "El parámetro 'edad' es requerido");
    return;
}

// 2. Parsear con manejo de excepción
int edad;
try {
    edad = Integer.parseInt(edadStr.trim());
} catch (NumberFormatException e) {
    response.sendError(HttpServletResponse.SC_BAD_REQUEST, "La edad debe ser un número entero");
    return;
}

// 3. Validar rango lógico (0–120)
if (edad < 0 || edad > 120) {
    response.sendError(HttpServletResponse.SC_BAD_REQUEST, "Edad fuera de rango permitido");
    return;
}

// 4. Lógica de negocio — ahora sí es segura
if (edad > 18) {
    registrarUsuario();
}

Mejoras aplicadas · Shift Left Security

  • 01 Validación de null y vacío antes de procesar
  • 02 Manejo de excepciones con try/catch
  • 03 Restricción de rango lógico (0–120)
  • 04 Mensajes de error sin exponer datos internos

Este ejercicio demuestra que incluso un fragmento de código pequeño puede esconder múltiples puntos de fallo. Aplicar Shift Left Security desde el primer sprint evita que estos problemas lleguen a producción.

06 — Reflexión Crítica

Lo que aprendimos


Analizar nuestro propio código con ojos críticos fue uno de los ejercicios más valiosos de este proceso. Lo que parecía una validación simple, verificar si alguien es mayor de 18, escondía tres vulnerabilidades reales que en producción podrían causar caídas del sistema o comportamientos inesperados.

Aprendimos que un desarrollador no puede limitarse a preguntarse ¿funciona el código? sino también ¿es seguro? Esas dos preguntas deben hacerse al mismo tiempo, desde el primer sprint

Shift Left Security no es solo una técnica; es un cambio de mentalidad. Nos enseña que la seguridad no se agrega al final, se diseña desde el principio. Como futuros ingenieros de software, adoptar esta filosofía nos hace mejores profesionales y nos prepara para construir sistemas en los que las personas puedan confiar.

"La seguridad no es una función del producto, es una propiedad que debe estar presente desde el primer commit."
